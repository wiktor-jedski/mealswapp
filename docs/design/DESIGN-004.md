## FILE: DESIGN-004.md
**Traceability:** ARCH-004

**Static aspects covered:** LPSolverWrapper, ConstraintBuilder, ObjectiveFunction, DiversityPenalizer, SolutionValidator, JobQueueManager, JobStatusTracker.

### 0. Static Aspect Responsibilities
- `LPSolverWrapper`: owns pure-Go serialization of validated LP models, invocation of the pinned native COIN-OR CLP executable as an OS child process, context deadlines, bounded output parsing, and solver result conversion. It does not link a CLP library through CGO and is never called from the Fiber API process.
- `ConstraintBuilder`: owns saved-diet target derivation, meal eligibility, protein, carbohydrate, fat, typed exclusion, bounded quantity, and multi-solution constraints.
- `ObjectiveFunction`: owns a lexicographic policy with calorie minimization as the primary objective and original-meal base-unit quantity as the secondary objective.
- `DiversityPenalizer`: identifies meals already present in the original diet, assigns their secondary coefficients without changing eligibility, and owns the canonical alternative-generation sequence.
- `SolutionValidator`: owns one immutable, UUID-indexed repository meal snapshot per generation call, shared saved-diet ID and owner validation at its public boundary, plus one validation/projection of each final solver result covering feasibility, macro tolerance, excluded UUIDs, selected-meal cardinality, canonical units, finite quantities, and authoritative quantity-weighted similarity.
- `JobQueueManager`: owns Redis-backed enqueue, reservation, retry, and worker execution.
- `JobStatusTracker`: owns per-user admission, job status persistence, polling responses, result TTL, and failure messages.

### 1. Data Structures & Types
- `type JobStatus = "queued" | "processing" | "completed" | "failed" | "cancelled"`
- `type OptimizationFailureCode = "failed_validation" | "solver_timeout" | "solver_infeasible" | "worker_crash"`; this is the complete persisted terminal-failure vocabulary. The Go representation has an unexported value, accepts only these legacy JSON strings, and rejects empty or unknown values during encoding and decoding.
- `interface SavedDietOptimizationRequest { dailyDietId: UUID; excludedMealIds: UUID[]; tolerancePercent: number }`
- `interface PersistedOptimizationAcknowledgement { jobId: UUID; status: "queued"; pollUrl: string; publicationState: "pending" | "published" }`; this is the sole durable acknowledgement shape. `publicationState` is internal persistence metadata and is never returned in the public acknowledgement data.
- `interface MacroTarget { protein: number; carbohydrates: number; fat: number }`
- `interface OptimizationJob { jobId: string; userId: string; request: DietOptimizationRequest; status: JobStatus; createdAt: time.Time; startedAt?: time.Time; finishedAt?: time.Time; error?: string }`
- `interface LPVariable { itemId: string; lowerBound: number; upperBound: number; caloriesPerUnit: number; proteinPerUnit: number; carbohydratesPerUnit: number; fatPerUnit: number; diversityPenalty: number }`, where `diversityPenalty` is `1` for an original meal and `0` otherwise, so the secondary objective is measured in repository base units (grams for solids and millilitres for liquids).
- `interface LPConstraint { name: string; lowerBound: number; upperBound: number; coefficients: map[string]float64 }`
- `interface ObjectiveFunction { coefficients: map[string]float64 }`
- `interface ObjectivePolicy { primary: ObjectiveFunction; secondary: ObjectiveFunction }`
- `interface DietAlternative { meals: MealQuantity[]; macros: MacroTarget; calories: number; similarityScore: number }`, where `similarityScore` is the quantity-weighted Jaccard ratio `Σ min(original[id], alternative[id]) / Σ max(original[id], alternative[id])` over the union of meal IDs after original quantities are converted to canonical `g`/`ml`. It is bounded to `0..1` and rounded to four decimal places; an identical quantity map is `1`, disjoint maps are `0`.

### 2. Logic & Algorithms (Step-by-Step)
1. API submission parses and validates the request before hashing. The canonical hash is calculated from the parsed non-nil diet UUID, validated one-decimal tolerance, and the exclusion UUID set sorted in ascending canonical UUID order. Duplicate exclusions are rejected, while JSON member order, insignificant whitespace, UUID letter case, and exclusion order do not affect identity. The validated idempotency-key string owns its bytes and cannot alias a reusable HTTP request buffer.
1. The user-scoped durable idempotency row is the cross-controller authority; no controller mutex serializes submissions. A typed acknowledgement is first claimed with `publicationState=pending`. Concurrent claim losers load that exact acknowledgement. Once publication is confirmed, `publicationState=published` is persisted and every exact replay returns the original `202`, `Location`, job ID, queued status, and poll URL without checking current entitlement, diet, job/result TTL, admission, job store, or queue and without side effects. A changed canonical request returns `409`.
1. Only a `pending` acknowledgement enters repair. Repair checks current entitlement and reloads the exact diet under the authenticated owner before reacquiring the same user's slot with `CountRate=false`; denial, deletion, owner mismatch, cancellation, or dependency failure publishes nothing. A repairer that newly acquires the slot re-reads the durable acknowledgement before side effects: if another controller already persisted `published`, it releases its fresh slot and returns that acknowledgement. It then idempotently saves the original job ID and calls the queue's cross-instance idempotent enqueue. An existing queue marker transfers a newly acquired slot only while the authoritative job remains `queued` or `processing`; a preserved terminal job leaves the slot controller-owned so deferred cleanup releases it. If the job becomes terminal after that ownership check, the worker's owner-scoped release removes the same job reservation. The repairer finally marks the acknowledgement `published`. This permits retry after a crash or queue failure without accepting stale or unauthorized input or stranding capacity after concurrent publication and terminal release.
1. For a request without a durable claim, API submission validates current entitlement and reloads the owned diet, then uses Redis `SET NX` to reserve the authenticated user's single active-job slot. A fixed-hour Redis counter admits at most 10 newly accepted jobs; rejected admission creates no idempotency row, job record, or stream entry. Different users and controller instances never share an in-process lock. Enqueue only the server-created job ID with `XADD`; return `202 Accepted` after publication confirmation.
1. A dedicated worker reserves deliveries through one `XREADGROUP` consumer group, reclaiming abandoned pending entries with `XAUTOCLAIM`, and loads the authenticated owner's exact non-nil saved diet from ARCH-005. The returned diet ID and owner must exactly match the request. Empty diets fail safely. The optimization stream and every queue-owned enqueue, attempt, lock, and terminal-marker key carry the same non-empty Redis hash tag (`{queue-v1}` by default), so the atomic multi-key scripts execute on both standalone Redis and Redis Cluster without `CROSSSLOT`; a custom stream without a hash tag is invalid configuration. Queue Lua is shipped as non-empty embedded resources and executed through `redis.Script`, using cached `EVALSHA` with automatic `EVAL` fallback after `NOSCRIPT`.
1. Eligible candidates are active repository meals with an explicit `solid` or `liquid` physical state and an available, valid normalized per-100 basis. Solid coefficients and quantities use grams; liquid coefficients and quantities use millilitres. The repository query requests only supported physical states, then the builder filters candidates whose normalized basis is unavailable. A candidate whose normalized protein, carbohydrate, and fat values convey zero objective information is also filtered when unrelated to the original diet. An unusable or zero-information original-diet meal fails the job because every original entry contributes to the authoritative target. Explicit UUID exclusions are omitted from the model, including when they refer to an original meal; original meals otherwise remain eligible.
1. Build one nonnegative LP variable per eligible candidate, bounded in the LP `Bounds` section to at most `10,000` grams or millilitres. This optimizer recommendation ceiling reuses the application's bounded physical-quantity policy and is intentionally stricter than the persistence ceiling for user-authored saved diets. Variable bounds are not duplicated as row constraints.
1. Derive Protein, Carbohydrate, and Fat targets only from the current server-owned saved-diet entries after canonical unit conversion, then create constraints using those targets and the client-selected tolerance band. All-zero authoritative targets fail before solving so the empty LP optimum cannot conflict with publication validation. Standalone client-authored macro targets are deferred to Phase 10.
1. Build and solve a lexicographic objective policy. First minimize total calories. Validate that solution, calculate its finite calorie value, and add that value as an exact equality constraint. Then minimize the summed grams/millilitres assigned to original-diet meals. A secondary solve is skipped when all secondary coefficients are zero. Because diversity is never numerically added to calories, it cannot overturn calorie ordering; it only selects among solutions at the fixed primary optimum. Every retained objective field is serialized as the objective of one of these solver passes. The secondary range is bounded by `0..(10,000 × eligible original variables)` base units, so no scaling constant or mixed-unit weighted sum is required.
1. Apply one hard 30-second deadline to repository loading and all alternative-generation solver attempts for the job. Run each pinned native `clp` invocation with `exec.CommandContext` under that shared context and retain the wrapper's 30-second per-process ceiling as a defensive upper bound. The wrapper writes a private per-job LP file and solution file, then unconditionally removes the temporary directory.
1. Before generation, copy repository meal values and their nested slices once, reject nil/empty/duplicate-ID snapshots, sort the detached slice by meal UUID, and build one read-only UUID index shared by model construction and publication validation. Caller mutation after this boundary cannot change a running generation. For every primary, secondary, and final solver assignment, remove signed zero residue within `1e-9 × max(1, compared magnitudes)` and reject unknown IDs, material negative values, non-finite values, and exact-model violations. Evaluate objective and constraint coefficient maps in ascending meal-UUID order with that same scale-aware tolerance. The final assignment is then validated and projected exactly once from the shared index: selected meals must number `1..100` as required by OpenAPI, quantities may not exceed `10,000` g/ml, macro tolerance must hold, excluded UUIDs must be absent, and calories must be finite.
1. Generate up to 3 alternatives with a deterministic bounded meal-set heuristic: only after canonicalization, exact-model validation, repository validation/projection, and duplicate rejection does the generator atomically add a result to the seen set, prior-solution constraints, and output. Projection recalculates macros/calories and then calculates the authoritative quantity-weighted Jaccard `similarityScore` from the same immutable meal snapshot and canonical quantities; no solver or client score is accepted. After each accepted solve, hard-exclude the selected meal with the greatest base-unit quantity (ties use ascending meal UUID) from the next solve. Every later alternative must therefore change the selected meal-ID set; quantity drift over an unchanged set is never distinct. A duplicate that also violates the resulting exclusion model is a validation failure and is never retried. The attempt budget is three times the capped requested result count. Exhausting it returns already validated alternatives without error; a later solver, context, model, duplicate, or projection failure returns already validated alternatives with its safe terminal error. No partial result is retained until all acceptance stages for that result pass. This heuristic deliberately trades completeness for a small continuous LP without binary selection variables.
1. Store completed or failed result in Redis with 1-hour TTL and update job status for polling. Existing retained terminal JSON remains compatible because the four approved codes keep their string representation. Decode rejects historical/foreign empty, `queue_unavailable`, `result_expired`, or unknown failure codes instead of projecting them. Failed records require the canonical safe message for their code; non-failed records cannot carry a failure.

### 3. State Management & Error Handling
- `queued`: accepted but not yet started.
- `processing`: worker owns the job; poll returns progress metadata only.
- `completed`: result exists and can be returned until TTL expires.
- `failed_validation`: terminal after worker-side request/model/result validation; publish the canonical safe message without validation diagnostics. Pre-enqueue HTTP request validation remains an ordinary 4xx envelope, not a persisted job failure.
- `solver_timeout`: mark failed after the single 30-second whole-job deadline and include partial alternatives if any passed validation; use a separate bounded finalization context so the terminal state can be published before ownership visibility expires.
- `solver_infeasible`: mark failed with a user-safe message that no combination matches the targets.
- `worker_crash`: terminal only after the queue's three delivery attempts are exhausted for an unknown worker/solver failure; publish a generic safe message. Before exhaustion, unknown failures remain pending and retryable.
- `queue_unavailable`: return retryable 503 from submission or leave an owned delivery pending when publication infrastructure fails; it is not a persisted terminal failure code and the worker does not run synchronously.
- `result_expired`: return retryable 410 only when an owned result TTL marker is found; it is not a persisted terminal failure code. Other users receive indistinguishable not-found.
- `optimization_in_progress`: return 429 with `Retry-After` when the authenticated user already owns another queued or processing job.
- `optimization_rate_limited`: return 429 with `Retry-After` after 10 newly accepted jobs in one fixed UTC hour; exact idempotency repair does not increment the counter.
- Every optimization 429 uses the shared retryable error envelope and a required positive integer `Retry-After` header. Fractional durations round up to the next whole second so callers never retry before admission permits it.
- Redis visibility timeout longer than the 30-second solver deadline returns a pending delivery through `XAUTOCLAIM`; `XACK` occurs only after the processor deliberately returns `PublishedCompleted` or `PublishedFailed`, or after exhausted-retry handling returns `PublishedFailed`. Missing or contradictory publication leaves the delivery pending. Completed and failed ACK entry points are distinct, and a terminal marker cannot change value.
- Finalization checks the existing terminal value, performs `XACK`, writes the terminal marker only for a pending delivery (or accepts an existing matching marker), and performs `XDEL` in one script. `XACK=0` is successful only when the same terminal marker already exists; otherwise it creates no marker and removes no entry. Duplicate/malformed cleanup performs `XACK` and `XDEL` in one separate script, where repeated zero-ack/zero-delete execution is idempotent.
- Processing-lock release uses the owner-token script under a cancellation-detached 100-millisecond deadline. Cleanup failure cannot replace the processing result and emits only bounded `optimization_queue_cleanup_total{outcome="failed"}` telemetry without job, entry, user, or payload fields.
- Queue statistics treat consumer-group lag as waiting work and the pending-entry list as owned work. `OldestQueuedAge` is derived from the first stream entry strictly after the group's Redis-reported last-delivered ID, so pending entries are excluded. `OldestPendingAge` is the greatest Redis-reported pending idle duration, not an age recomputed from a stream ID. Empty populations report zero; a waiting stream ID ahead of the application clock is clamped to zero.
- A live manager treats only Redis `NOGROUP` as evidence that its cached bootstrap state is stale. It clears that local state, idempotently recreates the stream/group at `0`, and retries the failed reserve/reclaim command once, recovering entries that survived group deletion and new entries published after non-persistent Redis restart. Authorization, connectivity, script, and all other Redis errors remain queue-unavailable failures and never trigger permissive recovery. Concurrent bootstrap/recovery is serialized per manager and `BUSYGROUP` remains idempotent success.

Cancellation is not a persisted failure code. Expiry of the worker-owned 30-second processing deadline while the parent worker remains live is `solver_timeout` and uses a bounded cancellation-detached finalization context. Parent-context cancellation means worker shutdown: return `context.Canceled`, publish no terminal result, release no admission ownership, and leave the stream delivery pending for reclaim/retry. A cancellation-like dependency or solver error while the parent is live is likewise retried; if it remains unknown through three deliveries, only then does terminal handling publish `worker_crash`. The frontend's local `AbortController` cancellation only stops polling and never changes server job state.

Each retained terminal code has one producer and consumer path: validation errors and solver infeasibility/timeout are produced by worker classification; `worker_crash` is produced by exhausted queue retries; Redis validates/decodes all four; HTTP polling projects all four; generated TypeScript and the optimization client accept exactly all four; the UI maps each to bounded text. Queue outage and expiry remain separately produced/consumed HTTP operation errors.

Alternative generation has a separate partial-result rule: an accepted alternative is immutable and remains publishable in the failed result when a later attempt returns a safe terminal error. Attempt-budget exhaustion is a successful partial completion. Invalid data from the current attempt is never appended, and failure before the first accepted alternative returns no alternatives.

The worker performs a bounded startup `clp -version` check and fails readiness unless the packaged executable reports the supported pinned version. The dedicated `linux/amd64` optimizer image contains the CGO-disabled Go worker and the checksum-pinned official CLP `1.17.11` Ubuntu 24 release artifact on an Ubuntu 24.04 runtime; CLP is a child process inside that container, not a separate network service. Model IDs and constraint names are mapped to generated solver names; they never become filenames or subprocess arguments. CLP stdout, stderr, and solution files are bounded and control-character sanitized before diagnostics are retained.
