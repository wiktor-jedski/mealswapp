# Phase 03.1 UAT: Account Flow Repair Gate

## Scope

Phase 03.1 covers repair tasks `107`-`114`. The repair pass wires the real API
bootstrap to Phase 03 repositories and services, tightens auth route policy,
adds route-level validation, normalizes consent versions, reconciles the profile
route surface, preserves typed empty custom-item exports, and completes account
deletion orchestration evidence.

The production bootstrap now mounts CSRF, registration, login, logout, refresh,
verification, password reset, OAuth, profile, saved-data, export, account
deletion, disclaimer, health, and readiness routes through real repository-backed
dependencies. OAuth routes fail closed until Google or Apple provider credentials
and callback exchange are configured.

## Project-Owner Checks

1. Start local dependencies with `bash scripts/start-services.sh`, then run
   migrations and `cd backend && go run ./cmd/api`.
2. Register and login with the current development consent versions; confirm
   local HTTP receives `mealswapp_access` and `mealswapp_refresh` HttpOnly
   cookies.
3. Fetch `/api/v1/auth/csrf-token`, delete the authenticated account with
   `DELETE /api/v1/account`, and confirm the response contains a pending
   deletion request and clears auth cookies.
4. Attempt login, profile, and export after the deletion request; confirm they
   reject the account/session.
5. Request `/api/v1/auth/oauth/google/start`; confirm the route is mounted and
   fails closed until provider configuration is added.
6. Download JSON and CSV exports for an active account and confirm `customItems`
   is present as an empty typed section.

## Known Deviations

- Backend internal coverage remains below 100% and is accepted under the
  documented Phase 03 coverage deviation in `docs/implementation/04_OPEN.md`.
  The Phase 03.1 aggregate check explicitly accepted that documented deviation.
- Redocly explicitly ignores `operation-2xx-response` for the OAuth start
  route because its successful provider-start response is a `302` redirect
  rather than a `2XX` response.
- Production OAuth provider credentials and callback exchange remain a
  project-owner/deployment action before enabling live Google or Apple login.
- Production key loading still needs the documented Secret Manager-backed loader
  before deployment; local development uses `MEALSWAPP_LOCAL_SECRET_KEY` or the
  development fallback.
- Privacy-law review is still required before production for the pseudonymous
  deletion receipt fields and provisional three-year retention period.
