## Scope

Phase 06.01 covers tasks `177`-`190`. Tasks `177`-`187` repair the Phase 06
checkout acceptance blocker by adding a real frontend authentication surface,
cookie-backed session projection, consent UI, authenticated
checkout guarding, authenticated Search and Subscription navigation, browser
workflow coverage, aggregate validation, and ARCH-018 SWE.5 integration
obligations. Task `188` is this acceptance document, task `189` refreshes the
completed-phase HTML report screenshots, and task `190` adds the targeted
hand-held UI repairs for the composed auth/search/subscription surface.

The repair supersedes the blocked Phase 06 checkout acceptance path that
required curl, manual cookies, or local development auth shortcuts before a
project owner could start checkout from the webapp. Phase 06.01 acceptance must
start from the real frontend sign-in/register surface and an HttpOnly-cookie
browser session. Local development auth shortcuts are test-only support and do
not satisfy checkout acceptance.

The implemented surface follows `docs/architecture/ARCH-018.md` and
`docs/design/DESIGN-018.md`:

- `AuthView`: sign-in/register surface, mode switching, authenticated summary,
  and handoff back to the invoking workflow.
- `RegisterView` and `ConsentGate`: email/password registration, password
  confirmation, current Privacy Policy and Terms of Service consent selection,
  stale-consent recovery, duplicate-email feedback, and successful session
  handoff.
- `LoginView`: email/password login, generic invalid-credential feedback,
  lockout and rate-limit retry metadata, duplicate-submit prevention, password
  clearing, and protected-action continuation.
- `AuthSessionStore`: frontend-safe `unknown`, `anonymous`, `authenticated`,
  `expired`, `locked`, and `error` state derived from server responses without
  storing access tokens, refresh tokens, password text, or raw CSRF secrets.
- `AuthApiClient`: generated-contract calls for CSRF, registration, login,
  logout, profile/session probing, OAuth return refresh, entitlement refresh
  coordination, and OAuth provider start URLs.
- `OAuthEntryPoint`: Google OAuth provider entry with the accepted `Google`
  label, fail-closed unavailable provider handling, and callback-return session
  refresh within the modal auth surface.
- `AuthenticatedActionGuard`: anonymous, expired, locked, or unverified users
  receive sign-in/register guidance before protected actions; successful
  authentication allows the queued protected action to retry once from the real
  browser session.
- `ComponentStyles`: hand-held sidebar, registration capture, and
  Search/Subscription navigation fixes keep mobile layouts free of horizontal
  scrolling, text overlap, clipped controls, inaccessible account navigation,
  and broken Search return flow.

## Checkout Acceptance Path

Phase 06 checkout acceptance is now started through the webapp:

1. Open the frontend webapp.
2. Use the visible auth surface to register or log in.
3. Confirm the browser session is established by server-backed session/profile
   state and HttpOnly cookies, not by manually setting cookies or injecting a
   development-only identity.
4. Navigate to the authenticated Subscription view.
5. Start monthly or annual Stripe-hosted Checkout from the billing controls.

This path updates and supersedes the blocked Phase 06 acceptance path. Curl
setup, manual cookie insertion, mocked local identity state, and local
development auth shortcuts may remain useful for developer tests, but they do
not prove project-owner checkout acceptance and must not be used as acceptance
evidence.

## Project-Owner Checks

### Integration Checks

1. Start PostgreSQL and Redis with `bash scripts/start-services.sh`.
2. Run migrations from `backend/` with
   `GOCACHE=$PWD/.go-cache GOMODCACHE=$PWD/.go-mod-cache go run ./cmd/migrate up`.
3. Start the API with local Stripe sandbox settings:
   `GOCACHE=$PWD/.go-cache GOMODCACHE=$PWD/.go-mod-cache go run ./cmd/api`.
4. From `frontend/`, run
   `BUN_TMPDIR=$PWD/.bun-tmp BUN_INSTALL=$PWD/.bun-install bun run dev`.
5. Open the served webapp in desktop and mobile-width browser viewports.
6. Confirm anonymous Catalog Search works before login and does not require
   account creation.
7. Confirm anonymous Subscription navigation or checkout intent opens
   sign-in/register guidance and does not call `/api/v1/billing/checkout`.

### Terms of Service Medical Information

1. Open the Terms of Service view.
2. Confirm it states that Mealswapp does not provide medical advice, diagnosis,
   or treatment and directs users to a qualified healthcare professional.
3. Confirm the auth modal does not duplicate this content. The About-section
   placement remains planned for the future About implementation.

### Register and Consent

1. Open the SearchShell auth modal in register mode.
2. Try to register without accepting the current Privacy Policy and Terms of
   Service versions. Confirm submission is blocked.
3. Accept the current consent versions, enter matching passwords, and register.
4. Confirm a successful registration creates an authenticated browser session
   projection and clears password fields.
5. Confirm duplicate-email feedback offers login mode without exposing stored
   password text or writing PII/password values to browser storage.
6. Confirm stale-consent responses refresh the current versions and require
   re-acceptance.

### Login and Logout

1. Open the auth surface in login mode.
2. Submit invalid credentials and confirm generic feedback that does not reveal
   whether the account exists.
3. Confirm lockout or rate-limit responses show safe retry timing when provided.
4. Confirm duplicate submissions are disabled while login is in progress.
5. Log in successfully and confirm Search and Subscription account navigation is
   available.
6. Log out and confirm authenticated UI state clears while anonymous Catalog
   Search remains usable.

### Search and Subscription Navigation

1. As an anonymous user, run Catalog Search and confirm it remains available.
2. As a logged-in user, confirm the sidebar shows `Search` and `Subscription`
   links.
3. Run a Catalog Search, open `Subscription`, then return to `Search`. Confirm
   the previous search state remains visible.
4. Confirm the Subscription view is separate from the primary Search surface and
   is available only through authenticated navigation.
5. Confirm anonymous users do not see an authenticated-only Subscription view or
   Subscription sidebar link.
6. Confirm keyboard focus reaches Search and Subscription navigation controls,
   and mobile sidebar navigation remains usable.

### Entitlement and Checkout

1. As a logged-in user, open Subscription and confirm authenticated entitlement
   status renders from the billing/entitlement API, including active tier,
   status, allowed modes, usage remaining, trial expiry when present, and billing
   recovery state when present.
2. Start monthly Stripe-hosted Checkout and confirm the application requests a
   server checkout session only after authentication.
3. Start annual Stripe-hosted Checkout and confirm the same authenticated path is
   used.
4. Confirm the browser is redirected only to the hosted checkout URL returned by
   the server.
5. Confirm the application UI never renders or submits PAN, CVC, CVV, card
   number, expiry, or raw payment-card fields. Card entry belongs only to
   Stripe-hosted Checkout.
6. Cancel an auth prompt before checkout and confirm the queued checkout action
   is cleared.
7. Authenticate after a guarded checkout intent and confirm the queued protected
   action retries once from the established HttpOnly-cookie session.

### OAuth and Session Recovery

1. Use the Google OAuth entry action and confirm it routes through the generated
   backend provider start URL without provider secrets in frontend state. The
   visible provider label is the accepted product copy `Google`.
2. Confirm Apple OAuth is unavailable or hidden until a later phase enables it.
3. Return from OAuth and confirm the app refreshes session/profile and
   entitlement state through backend calls. URL parameters alone must not mark
   the user authenticated.
4. Simulate an expired session and confirm protected Subscription navigation
   clears frontend-safe auth state, shows sign-in guidance, and preserves
   anonymous Catalog Search.
5. Confirm storage failures degrade to in-memory frontend state without breaking
   cookie-backed authentication.

### Responsive and Accessibility Checks

1. Run the focused Playwright auth, billing, and search workflows for desktop
   and mobile projects.
2. Confirm auth forms, consent controls, sidebar account
   links, Search view, Subscription view, and checkout actions fit without
   overlap at mobile widths.
3. Confirm keyboard-only operation reaches login/register fields, consent
   checkboxes, OAuth entry actions, logout, Search, Subscription, and checkout
   controls.
4. Confirm visible focus is present and controls have accessible names.
5. Confirm axe checks report no serious or critical violations on the auth
   surface and composed auth/search/subscription workflows.
