# Phase 07.01 UAT: Daily Diet and Optimization Remediation

## Scope

Phase 07.01 covers Tasks `213`-`237`. It closes the Phase 07 review findings
across the Daily Diet and optimization response contracts, canonical quantity
units, durable idempotency, native CLP process and parser boundaries, LP model
and alternative generation, Redis Streams ownership and recovery, frontend
runtime decoding and state coordination, safe errors and retry behavior,
observability, integration verification, aggregate quality, and acceptance
documentation.

The phase retains the Phase 07 product boundary: an authenticated, entitled
user optimizes a persisted Daily Diet; the API returns an asynchronous `202`
acknowledgement; a dedicated worker invokes CLP; and the browser polls for up to
three validated alternatives. Standalone macro-target generation and optional
WebSocket notification remain outside this phase.

The architecture and design sources are:

- [ARCH-004 Linear Programming Optimizer](../../architecture/ARCH-004.md),
  including the asynchronous API/Redis/worker/native-CLP boundary;
- [DESIGN-001 SearchView](../../design/DESIGN-001.md), covering browser search,
  Daily Diet selection, and UI state;
- [DESIGN-004 optimizer design](../../design/DESIGN-004.md), covering
  `LPSolverWrapper`, `ConstraintBuilder`, `ObjectiveFunction`,
  `DiversityPenalizer`, `SolutionValidator`, `JobQueueManager`, and
  `JobStatusTracker`;
- [DESIGN-005 UnitConverter](../../design/DESIGN-005.md), covering canonical
  physical quantity units and the isolated recipe-serving conversion;
- [DESIGN-008 SavedDataRepository](../../design/DESIGN-008.md), covering
  user-scoped Daily Diet persistence;
- [DESIGN-014 MetricsCollector](../../design/DESIGN-014.md), covering bounded
  queue, worker, dependency, latency, and age telemetry; and
- [DESIGN-017 ErrorMessageMapper and RetryManager](../../design/DESIGN-017.md),
  covering safe client errors and deliberate retry behavior.

The principal software requirements are `SW-REQ-006` (multi-meal Daily Diet),
`SW-REQ-021` (LP macro matching), `SW-REQ-022` (calorie-primary objective),
`SW-REQ-023` (best-effort meal diversity), `SW-REQ-030` (at most three
alternatives), `SW-REQ-036` (recipe per-unit conversion), `SW-REQ-042`
(entitlement/admission boundary), `SW-REQ-043` (cross-user privacy),
`SW-REQ-080` (two-second P95 target), and `SW-REQ-082` (capacity/scalability).
Their integration mapping is maintained in
[ARCH-004 integration verification obligations](../../testing/integration/ARCH-004-obligations.md).

## Project Owner Acceptance Tests

Use non-production test accounts and fixture meals. Start PostgreSQL and Redis,
apply migrations, run the API and dedicated worker with supported CLP
`1.17.11`, and run the frontend. Do not store cookies, CSRF tokens,
idempotency keys, request bodies, or personal data in evidence.

### 1. Durable Daily Diet create replay and canonical units

Steps:

1. As an authenticated user, create a Daily Diet with at least two solid and/or
   liquid meals using the UI.
2. Simulate a lost create response, then repeat the same intent with the same
   idempotency key.
3. Replace and then delete the Diet before replaying the original create again.
4. Reuse the same key with a changed body.
5. Try supported metric and imperial units and attempt `serving`, unsupported,
   and cross-basis units at saved-Diet boundaries.

Accept when:

- one write occurs and every exact replay returns the immutable original `201`
  response, even after replacement/deletion;
- changed-body key reuse returns a safe `409` without another write;
- server-derived macros and ordering are authoritative; and
- `g`, `ml`, `oz`, and `fl_oz` work in their physical basis while `serving`,
  unsupported, and cross-basis saved-Diet values fail safely.

### 2. Authoritative Daily Diet state and selection

Steps:

1. Load multiple saved Diets, select one, and enter and leave Daily Diet
   Alternative mode through Catalog and Substitution modes.
2. Replace quantities while selection changes, trigger a failed replacement,
   and reload.
3. Delete selected and unselected Diets, including while a load is delayed.
4. Log out and sign in as a different account while a read is pending.

Accept when:

- pending edits never appear with stale authoritative macros and cannot be
  optimized;
- confirmed writes are not lost or resurrected by later reads;
- UI selection, optimization input, and emitted search request always identify
  the same current Diet; and
- logout/account change clears and aborts prior-user Diet and selection state.

### 3. Normalized optimization submission replay and repair

Steps:

1. Submit a selected Diet and capture the `202`, `Location`, job ID, status, and
   poll URL.
2. Replay semantically equivalent JSON with reordered exclusions and UUID case
   differences using the same key.
3. Reuse the key after Diet deletion/change, entitlement change, and result
   expiry; separately simulate a failed publication before retry.
4. Submit a genuinely changed tolerance or exclusion set with the same key.
5. Submit concurrently as two different users.

Accept when:

- a published exact replay returns the byte-equivalent acknowledgement with no
  entitlement, Diet, admission, job-store, or queue side effect;
- a pending-publication repair revalidates ownership/input and publishes once;
- changed canonical input conflicts with `409`;
- each user progresses independently with one owner-scoped job; and
- cross-user polling reveals no job or ownership data.

### 4. Solver correctness and distinct alternatives

Steps:

1. Optimize a Diet with known repository macros and a feasible tolerance.
2. Use fixtures where calorie order and original-meal overlap would prefer
   different solutions.
3. Add exclusions and include metric solid and liquid candidates.
4. Exercise one, two, and three alternatives; then force a later solve failure
   after one valid result.
5. Exercise infeasible, malformed/duplicate solver output, tiny negative
   residue, materially negative values, and timeout.

Accept when:

- every published alternative matches authoritative protein, carbohydrates,
  and fat tolerance after canonical conversion;
- calorie minimization remains primary and diversity is only secondary;
- retained alternatives have distinct meal-ID sets and total count is `1..3`;
- exclusions, quantity/cardinality bounds, and similarity `0..1` are valid;
- malformed/duplicate/materially negative output is never published; and
- a valid partial result survives an approved later failure while timeout and
  infeasible states expose only canonical safe failures.

### 5. Queue ownership, retry, finalization, and recovery

Steps:

1. Process duplicate stream deliveries and concurrent consumers.
2. Cancel a worker-owned delivery, reclaim it, and exercise three genuine
   failed attempts.
3. Inject a malformed/non-canonical UUID delivery.
4. Delete the consumer group and restart Redis using the test fixture, then
   enqueue and process new work.
5. Inject missing terminal publication and cleanup/finalization failure.

Accept when:

- duplicate or lock-miss deliveries do not consume attempts or duplicate
  authoritative work;
- cancellation leaves work reclaimable and three owned attempts precede one
  durable terminal failure;
- malformed deliveries are acknowledged/deleted without processing;
- final acknowledgement requires terminal publication and atomic cleanup;
- group/data/restart loss recovers through bounded idempotent bootstrap; and
- authorization/connectivity/cleanup contradictions fail closed and remain
  bounded and observable.

### 6. Safe errors, retry policy, expiry, and identity reset

Steps:

1. Return malformed envelopes, unsafe request IDs/messages, extra fields,
   backend diagnostics, and wrong successful statuses to both clients.
2. Exercise network ambiguity, queue outage, poll timeout, infeasible,
   validation, auth, entitlement, cancellation, worker failure, and expiry.
3. Edit tolerance/exclusions before retrying.
4. Unmount/remount while submitting and polling, then log out and switch
   account while delayed work is active.

Accept when:

- malformed or unsafe payloads never reach stores or rendering;
- messages contain no stack, SQL/Redis/provider name, URL, credential, body, or
  unbounded request ID;
- ambiguity reuses its key, known-job polling resumes deliberately, fresh
  intent rotates its key, and non-retryable failures expose no blind retry;
- expired results create fresh intent and disclose no cross-user ownership;
- remount resumes or resets deliberately without permanent busy state; and
- logout/account switch clears jobs, results, errors, keys, selected Diet,
  polls, timers, listeners, and late prior-user completion.

### 7. Desktop, mobile, keyboard, themes, and accessibility

Steps:

1. Run create/replay, replace/select, submit/poll/retry, infeasible, timeout,
   queue-failure, logout, and account-change workflows in desktop and mobile
   Chromium.
2. Repeat the maintained theme scenarios in light and dark mode.
3. Complete the controls using keyboard only and inspect focus order and focus
   visibility.
4. Run axe and inspect screenshots for stale, unsafe, clipped, or overflowing
   state.

Accept when:

- controls and status content remain usable without horizontal overflow or
  clipping in both viewports and themes;
- keyboard operation reaches every required action with visible focus;
- axe reports no serious or critical violations; and
- screenshots show only current authoritative Diet and optimization state.

### 8. Degraded dependencies, observability, and capacity evidence

Steps:

1. Stop or restart Redis during submission, polling, waiting/pending age
   measurement, and group recovery.
2. Exercise queue publication failure, solver timeout, admission release
   failure, lock cleanup failure, retry exhaustion, and worker heartbeat loss.
3. Run concurrent unrelated submissions, same-key replay, and polling through
   `scripts/verify-phase0701-observability-capacity.py`.
4. In an approved non-production deployment, separately run the authenticated
   capacity profile with operator-managed credentials and fixture body.

Accept when:

- API work never falls back to synchronous solving and degraded submissions
  return bounded safe errors;
- queued and pending depths/ages match authoritative Redis state and never go
  negative;
- outcomes match final state and labels remain fixed-cardinality and free of
  identifiers, keys, bodies, meal data, and diagnostics;
- cleanup failures cannot block or replace the primary response;
- the deterministic fixture keeps submission/replay/poll checks below the
  documented two-second critical threshold; and
- any production `SW-REQ-082` claim is based on separate deployment load
  evidence rather than this local deterministic fixture.
